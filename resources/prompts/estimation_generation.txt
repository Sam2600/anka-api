# Rules
1. Use function_id format F001, F002, F003... in insertion order.
2. sheet2_features[i].function_id and sheet3_manhours[i].function_id MUST match 1:1 — every feature listed in sheet2 must have a corresponding dev_hours row in sheet3 with the same function_id.
3. Only propose dev_hours per feature in sheet3 — do NOT compute downstream phase costs (code review, prototype, etc.). Those are derived by the spreadsheet via fixed multipliers.
4. sheet5_team_stack[*].monthly_allocation.length MUST equal sheet4_milestone.total_months. Use values in [0.0, 1.0] representing fraction of a person-month.
4a. **`sheet4_milestone.total_months` is YOUR shortest effective timeline recommendation derived from the workload description — NOT a copy of the client-expected timeline.** Compute it from:
    - Total dev hours: `Σ sheet3_manhours[].dev_hours` (the bottom-up workload you yourself proposed)
    - Team capacity per month: `Σ (sheet5_team_stack[i].count × 160 × avg(sheet5_team_stack[i].monthly_allocation))`. Cap any single person's allocation at 1.0 — no heroics, no overtime.
    - Phase dependencies: requirements + architecture + basic_design must precede implementation_*; integration_test + system_test run after implementation. Allow modest parallelism across implementation streams (e.g., web and android can overlap; backend and frontend can overlap after architecture is locked).
    Aim for the **SHORTEST reasonable total_months that still respects realistic capacity + phase ordering**. Do NOT pad. Do NOT round up to match the client expectation. If your number differs from `Client-expected timeline (months)` above, state the delta in `reasoning` (e.g., "Recommended 5 months vs. client's 8 — feature set is small enough to compress" or "Recommended 11 months vs. client's 8 — the integration phase alone needs 3 months given the listed third-party APIs").
5. Use role names exactly as they appear in the {{ORG_ROLES_BLOCK}} below. Do not invent new role titles.
5a. Each sheet3_manhours row MUST carry a `role` field naming the role that will do the work (Backend Engineer, Frontend Engineer, QA Engineer, etc.). Match a role title from {{ORG_ROLES_BLOCK}} verbatim. Do NOT default everything to a single role — pick the appropriate one per feature. Account Director / Finance Manager rarely build features; reserve those for management/finance-flavoured work only.
6. Categories must be one of: "Web", "Mobile", "Integration", "Infrastructure".
7. Round dev_hours to whole numbers when ≥ 4, to one decimal otherwise.
8. Aim for 10–40 features. Too few misses scope; too many becomes noise.
9. Set confidence to "high" if a contract document is provided AND past won deals match this scope, "medium" if only one of those holds, "low" otherwise.
10. project_overheads — predict project-specific overhead line items based on the client and workload description. Each entry is a one-time or recurring expense the agency would incur DELIVERING this specific deal (NOT general company overhead). Examples to consider when relevant:
    - Travel & on-site visits (when the description implies client meetings, on-site discovery, or a regulated industry)
    - Specialised software licenses (CMS licenses, design software, monitoring tools tied to the stack)
    - Compliance / audits (HIPAA, SOC2, PCI, GDPR — if industry signals are present)
    - Translation / localisation (if multi-language is implied)
    - Third-party APIs / services (payment gateways, geolocation, analytics — only when implied by description)
    - Cloud infrastructure setup (significant cloud spend over the project — distinct from ongoing hosting)
    - Hardware / devices for testing (mobile devices, IoT hardware — only when implied)
    Costs are TOTAL project costs in the client budget currency. Include a short `reason` per item linking it to the deal context. Output 0–6 items — only ones the description genuinely supports. Do NOT pad with generic items.
