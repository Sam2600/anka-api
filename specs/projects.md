# anka-api — Projects Module Spec

## Models

### `Project`
File: `app/Models/Project.php`

**All fields:**
```
id              UUID PK
tenant_id       UUID FK → tenants
contract_id     UUID FK → contracts  (UNIQUE — enforces 1-to-1 with Contract; RESTRICT delete)
project_number  varchar(255) UNIQUE  auto: PRJ-101 via project_number_seq
name            varchar(255)
client          varchar(255)
budget_hours    decimal(10,2) default 0
consumed_hours  decimal(10,2) default 0  incremented by time entry approvals
status          enum  Not Started | On Track | At Risk | Over Budget | Completed  default: Not Started
start_date      date  nullable
end_date        date  nullable
created_at, updated_at, deleted_at
```

**Relationships:**
```
belongsTo: Contract
hasMany:   TimeEntry
```

### `TimeEntry`
File: `app/Models/TimeEntry.php`
```
id           UUID PK
tenant_id    UUID FK → tenants
project_id   UUID FK → projects  (RESTRICT delete)
employee_id  UUID FK → employees  (RESTRICT delete)
approved_by  UUID FK → users  (SET NULL)
task         varchar(500)
date         date
hours        decimal(6,2)  min 0.5, max 24
billable     boolean  default true
status       enum  Draft | Pending | Approved | Rejected  default: Draft
notes        text  nullable
approved_at  timestamp  nullable
```

## Project Status Values and Transitions

| Status | Meaning |
|---|---|
| `Not Started` | Created by win_deal(), no work begun |
| `On Track` | Active, burn rate within budget |
| `At Risk` | Close to budget limit (frontend guideline: >70% burn) |
| `Over Budget` | Exceeded planned hours (frontend guideline: >85% burn) |
| `Completed` | Delivery done |

**Transitions:** Any status can be set to any other via `PATCH /projects/{project}` — no enforced guard.

---

## Time Entry Status Values

| Status | Meaning |
|---|---|
| `Draft` | Created; not yet submitted |
| `Pending` | Submitted for approval |
| `Approved` | Approved; `consumed_hours` incremented |
| `Rejected` | Rejected; `consumed_hours` NOT affected |

## Business Rules

### Project
- Projects are created **exclusively** by the `win_deal()` stored procedure when a Deal is won.
- No `POST /projects` endpoint — do not add one without migrating the SP logic.
- `project_number` auto-generated by PostgreSQL sequence — never set manually.
- `consumed_hours` is incremented atomically when a TimeEntry is approved — never set directly unless correcting data.
- `contract_id` has a UNIQUE constraint — one project per contract, enforced at DB level.
- Updatable fields via `PATCH`: `status`, `name`, `budget_hours`, `end_date`, `consumed_hours`.

### TimeEntry
- Created with `status = 'Draft'` always — frontend cannot set an initial status.
- `hours` must be between 0.5 and 24.
- Approval runs in `DB::transaction()` with `lockForUpdate()` (pessimistic lock) to prevent double-increment of `consumed_hours`.
- Guard in approval: throws `Exception('Already approved')` if entry is already Approved — prevents duplicate increments.
- Approval increments `projects.consumed_hours` directly via `DB::table('projects')->increment(...)`.
- No "reject" endpoint exists — `status = 'Rejected'` can only be set by direct update (no dedicated route).

## API Endpoints

### Projects

#### `GET /api/projects`
- Filter: `contract_id`, `status`
- Pagination: `per_page` (default 50, max 200)
- Response: paginated `ProjectResource`

#### `GET /api/projects/{project}`
- Response: `ProjectResource`

#### `PATCH /api/projects/{project}`
- Allowed updates: `status`, `name`, `budget_hours`, `end_date`, `consumed_hours`
- Response: `ProjectResource`

#### `DELETE /api/projects/{project}`
- Soft delete
- Response: 204

### Time Entries

#### `GET /api/time-entries`
- Filter: `project_id`, `employee_id`, `status`, `date_from`, `date_to`
- Pagination: `per_page` (default 50, max 200)
- Ordered by `date DESC`

#### `POST /api/time-entries`
- Required: `project_id`, `employee_id`, `task`, `date`, `hours` (0.5–24)
- Optional: `billable` (boolean), `notes`
- Status always set to `Draft` by server
- Response: `TimeEntryResource`

#### `GET /api/time-entries/{time_entry}`
- Response: `TimeEntryResource`

#### `DELETE /api/time-entries/{time_entry}`
- Hard delete (no soft deletes on TimeEntry)
- Response: 204

#### `PATCH /api/time-entries/{time_entry}/approve`
- No request body
- Transaction + lockForUpdate
- Increments `projects.consumed_hours` by `hours`
- Sets `status = 'Approved'`, `approved_at = now()`
- Guard: throws 500 if already Approved
- Response: `TimeEntryResource`

## Links to Adjacent Modules

| Link | Status | Detail |
|---|---|---|
| Project → Contract (upstream) | ✅ EXISTS | `projects.contract_id` FK; `belongsTo(Contract)` |
| Project → Deal (indirect) | ✅ EXISTS | Via Contract → Deal chain |
| Project → Employee (via TimeEntry) | ✅ EXISTS | `time_entries.employee_id` FK |
| Project created from Contract signed | ⚠️ NOT IMPLEMENTED | No "Signed" gate; Project is auto-created with Contract in win_deal() |
| Team assignment on Project | ⚠️ NOT IMPLEMENTED | No direct team assignment on Project model; team is tracked at Deal level (hard_assignments) |

---

## What is Missing

- No `Rejected` endpoint for time entries — rejection can only be done by direct DB update or a PATCH to `status` (no dedicated route enforces the Rejected state).
- No team member assignment on the Project itself — only deals have `hard_assignments`. Once a project is running, who works on it is tracked implicitly via `time_entries`.
- No project start trigger — `start_date` is set manually, not automatically on first time entry.
